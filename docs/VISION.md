# Vision

## The category

> **Tailwind is a string vocabulary interpreted by Node. Chirpui is a Python vocabulary interpreted by Python.**

For a Python developer, that single fact is the category difference. It means the design system â€” its tokens, variants, slots, contracts, and rendered output â€” is reachable from the same interpreter running the app. The developer's own toolchain (`ty`, `pytest`, `pyproject.toml`, their IDE, their coding agent) can reason about the UI directly, without a parallel JS build, without round-tripping through a config language they don't use, without drift between what the designer declared and what the tests check.

Tailwind cannot enter this category without becoming a Python library and abandoning the utility-class paradigm. It will not. That leaves the category open.

## Why this matters for Python devs

Tailwind asks a Python developer to:

- Maintain a Node toolchain (`package.json`, `tailwind.config.js`, PostCSS) alongside their Python one.
- Express design decisions in a JavaScript configuration they cannot round-trip through `pyproject.toml`, `ty`, or their test suite.
- Test UI output by asserting on class-name strings â€” the same class-name strings an AI agent will hallucinate confidently.
- Build their own component layer on top of utilities, because Tailwind has no opinion on what a card is.
- Wire HTMX and Alpine themselves, hitting the same boost / select / form-reset pitfalls every project hits.

Chirpui collapses those into: write Python, render templates, run `uv run poe ci`. The *source of truth* lives where the developer already is.

## The flywheel

Five capabilities compound. Only one is hard; the rest follow.

1. **A type-checked component registry** (the one bet). Every macro exposes its slots, parameters, variants, `provides`/`consumes` keys, and HTMX support as a Python object. Importable, introspectable, versioned.
2. **Tokens in `pyproject.toml`.** Theme, spacing, typography, color tokens declared in Python-native config. CSS generated at install time. One edit round-trips through the whole stack.
3. **Contract verification as a verb.** `registry.verify(rendered_html, contract="dashboard")` returns a typed result. Dashboard maturity, accessibility contracts, composition contracts â€” all callable from tests or CI.
4. **HTMX- and Alpine-native defaults.** Boost-aware components, auto `hx-select="unset"` on forms inside boosted layouts, fragment islands, SSE / suspense primitives â€” problems solved once, inherited everywhere.
5. **Agent-groundable manifest.** The registry serializes to a machine-readable manifest an AI coding agent can cite, reducing UI hallucination to near zero. Docs regenerate from the same source. Shipped as `chirp_ui.manifest.build_manifest()` (Python API), `python -m chirp_ui.manifest --json` (CLI), and `site/public/chirpui.manifest.json` (site artifact, regenerated by `poe docs-build-all`). Schema: `chirpui-manifest@1`. See `src/chirp_ui/manifest.py`.

## What we are not

Not a utility-class expansion. Chirpui uses `stack()`, `cluster()`, `grid()`, `frame()`, `block()` as composition primitives; it does not ship `p-4 mx-auto rounded-lg` equivalents. The discipline is load-bearing â€” once a utility vocabulary exists, contracts and introspection erode, and we re-enter Tailwind's category at a disadvantage.

Not a React-replacement component system. Chirpui does not own state. It renders HTML and lets HTMX / Alpine handle interactivity. That is the shape that lets it remain a Python vocabulary.

Not a framework. It is an opinionated UI layer on top of `kida-templates` + `bengal-chirp`. Adoption is additive; any Chirp app can use none, some, or all of it.

## CSS architecture as a registry projection

The long-term CSS design is the one that makes the category win *sustainable*:
the stylesheet must be an output of the Python component registry, not a
parallel source of truth. The moment CSS classes, tokens, or override points
can drift from the registry, chirpui starts re-earning its correctness every
release the way Tailwind projects do.

Five rules define the shape:

1. **Tokens live in Python.** Spacing scale, color palette, motion curves,
   radii, elevations â€” declared in `pyproject.toml` or a `chirp_ui.tokens`
   module, typed, `ty`-checkable, diffable. CSS custom properties are
   generated from them at install/build time. A consumer edits one file; the
   CSS, the macro defaults, the test contracts, and the agent manifest all
   reflow consistently.

2. **Every class in the shipped CSS is registry-cited.** If `chirpui-card__X`
   is emitted by `card.html`, it appears in the card registry entry's
   `emits` set; if it's styled in `_card.css`, it's in that same set. Unused
   classes are deletable because the registry knows. Tests assert the three
   views agree. Agent manifests cite real classes, not plausible ones.

3. **The cascade order is public API.** One documented line:
   `chirpui.reset < chirpui.token < chirpui.base < chirpui.component <
   chirpui.utility < app.overrides`. Consumers don't fight specificity; they
   place overrides in `app.overrides` and win predictably. This is the
   CSS-layer analog of component override-by-slot.

4. **`@scope` envelopes mirror registry entries.** One partial per registry
   entry, each wrapped in `@layer chirpui.component { @scope (.chirpui-X) to
   (.chirpui-X .chirpui-X) { â€¦ } }`. The stylesheet's structure mechanically
   matches the registry's structure â€” reading a partial tells you what the
   registry entry knows, and vice versa. See
   `docs/plans/PLAN-css-scope-and-layer.md`.

5. **No utility class vocabulary, ever.** Composition primitives (`stack()`,
   `cluster()`, `grid()`, `frame()`, `block()`) stay as macros. Once a
   utility class ships, the registry ceases to be the single source of truth
   and cascade control becomes the consumer's homework â€” that is Tailwind's
   category, not ours.

What this means competitively: Tailwind cannot do 1â€“4 without becoming a
Python library. Open Props cannot do 4â€“5 without a component layer. Shadcn
inherits Tailwind's drift problem. Radix Themes inherits the React-toolchain
problem. The intersection â€” Python-native tokens, registry-cited classes,
HTMX-native components, `@scope`-organized CSS with a layered override API â€”
is uniquely reachable by chirpui.

The authoring format (`@scope` + `@layer` + native nesting + BEM API) is
instrumental. The categorical move is that every piece of it is a *projection
of the Python registry*. That projection is the asset competitors cannot
copy without adopting Python.

## The concession

The Tailwind ecosystem â€” plugins, Tailwind UI, Shadcn ports, landing-page generators, every designer's Figma plugin â€” is enormous and we will not have it. For prototyping a one-off marketing page, Tailwind is faster. For building a Python-native application whose UI must stay correct over years, chirpui wins by being reachable from Python.

We are choosing *correct, introspectable, small ecosystem* over *unbounded, opaque, large ecosystem*. That choice is the product.

## Decision lens for future PRs

When evaluating a change, ask:

- Does this preserve the property that chirpui is a Python vocabulary, not a string vocabulary?
- Does it make the registry richer, or does it grow a parallel source of truth?
- Can the test suite, `ty`, and a coding agent all reason about it?
- Does it inherit from a contract, or does it reinvent one?

If the answer to any is no, the change belongs somewhere else â€” or the design needs to bend until the answers are yes.
