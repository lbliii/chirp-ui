# Screen Pattern Promotion Ledger

Status: active promotion gate
Date: 2026-05-23

This ledger records repeated screen-level composition pressure. It exists so
Chirp UI can raise the taste floor without turning every attractive mock into
a public macro.

Promotion remains conservative: recipes precede APIs, and public vocabulary
requires registry, CSS, docs, generated output, tests, and stop-and-ask review.

## Promotion Ladder

| Level | Meaning | Allowed Output |
|---|---|---|
| Observation | One screen needed a local relationship, hierarchy, or state move. | Keep inside the screen recipe and record the pressure. |
| Repeated recipe | Two independent screens need the same semantic move. | Add or strengthen recipe guidance; no public macro. |
| Candidate | Three independent screens need the same move and existing primitives are awkward in the same way. | Write a promotion brief with source paths, browser proof, accessibility notes, and not-now alternatives. |
| Public surface | Stop-and-ask approves a component, primitive, token, profile, or macro plan. | Implement registry/CSS/manifest/docs/examples/tests/generated-output collateral together. |

Independent means distinct product situations, not two variants of the same
showcase fixture.

## Promotion Brief Requirements

Before a candidate becomes public vocabulary, write a brief that names:

- the source screens and routes,
- the exact repeated awkwardness in current primitives or components,
- why the candidate is semantic rather than visual shorthand,
- the chosen layer: primitive, component, pattern doc, shell recipe, token, or
  profile,
- accessibility and keyboard implications when interactive,
- responsive behavior at 320, 390, 768, and 1280 widths,
- generated-output and registry collateral if it becomes a component,
- docs, examples, and source-map updates,
- why recipe-only guidance is no longer enough.

## Stop-And-Ask Boundary

Stop before:

- adding a public screen macro,
- adding a public shell macro,
- adding a component variant, size, appearance, tone, or macro parameter,
- adding public typography role tokens,
- adding new theme packs or profile metadata,
- adding manifest/schema fields,
- changing cascade order,
- changing generated-output build order.

## Anti-Promotion Rules

Do not promote a pattern because:

- it made one static mock look better,
- the name sounds useful,
- a known design system has a similar component,
- local CSS could be hidden behind a macro,
- a planned archetype mentions the same word but has no fixture proof.

Promotion is justified only when repeated product situations prove that the
current Chirp UI path produces too much local workaround for a semantic job.
